01 — personas

Quem abre o app, e quando.

O mobile não substitui o web — ele complementa. Cada papel tem um momento de uso distinto: o GP checa durante o trajeto, o owner quer um pulso do portfólio entre reuniões, o cliente abre o link uma vez por sprint. Projetamos pro GP primeiro porque é quem mais volta.

Project Manager · GP

Maria, Diretora

35–45 · 5–15 projetos · Jira diário

★ Persona primária do mobile

  • Confere sprint antes de reunião de cliente
  • Recebe alerta de PR stale no Uber e decide ação
  • Lança horas de reunião no final do dia
  • Compartilha progresso no grupo do cliente
Owner · CEO

Ricardo, Fundador

40+ · visão de portfólio · foco em margem

  • Abre pra ver saúde geral (health by project)
  • Alerta de budget overflow leva ação imediata
  • Atribui/remaneja GP entre sprints
  • Exporta relatório executivo
Viewer · Cliente

Ana, Stakeholder

Read-only · uso episódico

  • Abre link do portal no mobile 1–2x por semana
  • Vê progresso e burndown do sprint atual
  • Não vê scorecard nem budget
  • Não entra em nenhum fluxo de escrita

04 — fluxos críticos

As quatro jornadas que precisam ser perfeitas.

Tudo mais pode ser wireframe funcional. Estas quatro jornadas recebem o maior polimento porque representam os momentos em que o mobile ganha ou perde o usuário: primeiro acesso, alerta em movimento, decisão de sprint, e lançamento de horas.

01

First-time login & onboarding

time-to-first-value < 60s

Usuário já tem conta criada no web. No mobile, login deve ser curto, push permission deve aparecer no momento certo (não na cara), e a primeira tela vista tem que ter valor imediato — o projeto mais ativo.

01WelcomeLockup + CTA "Entrar". Sign-up manda pro web.
→
02LoginEmail + senha. Opção "Lembrar" + biometria.
→
03Push permissionSoft ask primeiro ("quer receber alertas críticos?"), depois o prompt nativo.
→
04Org selectionSe >1 org, escolhe. Se 1, pula.
→
05PortfolioJá carregado com dados em cache da última sessão.
02

Push "PR stale 72h" → decisão

critical — < 3 taps

Fluxo mais importante do app. GP recebe push em movimento — tem que chegar na decisão em no máximo 3 taps e uma rolagem. Push tem deep link direto para o PR detail.

01Push notification"⚠ PR #187 parado há 72h — refactor(api): extract payment service"
→
02PR DetailCard, status, autor, idade, link card, link review.
→
03Ação rápida3 botões: "Abrir no GitHub", "Ping autor (Slack)", "Snooze 24h"
→
04ConfirmaçãoToast + card marca como "pinged" por 24h.

Manhã do GP — sprint health check

uso recorrente · 2–5 min

Uso mais frequente. GP abre de manhã pra saber onde estão os projetos. Pull-to-sync força update. Dados em cache garantem abertura instantânea mesmo offline.

Lançar horas manualmente

fim do dia · batch input

GP precisa lançar horas que não vieram do Tempo/Clockify — reuniões, discovery, briefing. Batch input em uma tela só, pré-preenche com projeto ativo, pra não virar fricção.

05 — push notifications

Seis alertas, nenhum spam.

Toda notificação tem um quick-action. Agrupadas por projeto na inbox. Resumo diário (digest) é opt-in e reduz o ruído pra quem acha 6 tipos muito.

07 — stack recomendada

React Native Expo, pelo mesmo motivo do resto.

O PRD lista React Native como escolha. Mantive, com Expo pra acelerar OTA updates e dev loop. Code sharing com web via packages/types — Zod schemas e TS types já compartilhados.

Recomendado — Expo + RN

Ecossistema moderno, TypeScript first, deploy fácil, OTA updates.

  • frameworkExpo SDK 52 · RN 0.76
  • navigationExpo Router (file-based, igual Next)
  • estadoReact Query (reuso do web)
  • authNextAuth JWT + expo-secure-store
  • pushExpo Notifications + APNS/FCM
  • tipografiaexpo-font · Inter + Fraunces
  • chartsVictory Native XL (burndown)
  • offlineReact Query cache + MMKV storage
  • shared@spravio/types (monorepo)

Alternativas

Dependendo de restrições.

  • rn bareSe precisar nativo custom (OCR, BLE). Build próprio.
  • flutterNão recomendo — força segunda linguagem (Dart), quebra code sharing.
  • pwaSó se mobile for priority baixa. Sem push iOS decente até 2024+.
  • nativoSwiftUI + Kotlin Compose. Qualidade máxima, dobro do custo.
08 — próximos passos

O caminho daqui até o TestFlight.

  1. Você valida este sitemap

    Me diz o que falta, o que sobra, o que muda de MVP pra v2. Discutimos escopo de cada tela antes de desenhar. 1–2 rodadas.

  2. Eu importo o repo do GitHub

    Leio os componentes React do web, extraio tokens reais (cores, radii, fontes da Tailwind config), e gero o design system mobile refletindo o que já existe. Garante consistência automática.

  3. Hi-fi das 5 telas principais (iOS + Android lado a lado)

    Portfolio, Sprint, PR detail, Inbox, Me. Cada uma em frame iOS + frame Android. Com tweaks ao vivo (cor, densidade, variantes de cada tela).

  4. Telas secundárias + estados

    Empty states, loading, erro, offline. Login, onboarding, modais de ação. Cobertura completa do MVP.

  5. Protótipo clicável (opcional)

    HTML interativo simulando navegação, push, swipe. Pra testar com usuários antes do dev começar.

  6. Handoff para Claude Code / dev

    Especificação pixel-perfect, tokens exportados, componentes mapeados. Dev começa Expo Router direto com tudo definido.
